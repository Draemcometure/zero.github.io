首先需要理解时序图：

这是一个游戏正常的表面流程，登陆—>进入游戏—>开始游戏—>根据游戏规则进行游戏—>游戏结束。那么我们就可以为这个游戏实现不同流程的区域，并且让他们相邻流程的耦合性更强，这样就需要使用面向对象的方式。

在这款纸牌游戏里，玩家选项无疑最为关键。我们为此制定如下规则：
下注环节，后下注者的下注金额不得低于前面下注者的金额；跟注时，金额需与之前下注者一致；若选择弃牌，已投入的资金将留在奖池中，后续 Allin 返钱时，弃牌者无法获得资金。比牌开始时，发起比牌的玩家需先进行下注。一旦有人选择 Allin，其他玩家仅能选择弃牌或者同样 Allin。

规则既定，接下来明确游戏结束的情形。实际上，仅存在两种情况会导致游戏结束：其一，所有玩家都选择 Allin；其二，历经所有比牌流程后，牌桌上仅剩下一名玩家。

好，下面我们先从较为简单的弃牌和 Allin 操作着手。

1.1.1.0 弃牌和 Allin

在进入这部分程序逻辑之前，我们需要清楚如何遍历每一位玩家。这里我们采用一种类似链表的方法来寻找下一位玩家。
我们将 start 设定为头指针，以此为起点寻找下一位玩家。为避免数组越界，采用取余的方式来循环遍历。需要注意的是，寻找到的下一位玩家必须仍处于游戏中。

1.1.1.1 弃牌

这里借助 Kotlin 特有的语法，以实时记录的方式来实现弃牌功能。
关于 gameOver () 函数，其原理是通过程序结束函数来判定游戏是否结束。因为我们清楚，若有三名玩家弃牌，那么剩余的那一位玩家便可赢得全部奖金。

1.1.1.2 ALL_IN

依据之前制定的规则，当有玩家选择 ALL_IN 时，其余玩家只能选择 ALL_IN 或者弃牌。所以，我们需要一个变量来标记是否有玩家率先选择 ALL_IN，若有，记录下其下标。
由于 ALL_IN 这一操作，当轮到下一位玩家操作时，其操作界面会发生变化。若已有玩家选择过 ALL_IN，那么该玩家只能在弃牌和 ALL_IN 中做选择。

1.1.1.3 跟注

为实现跟注功能，我们设置一个全局数据成员 LastBet，用于记录每次最小跟注的金额。该操作相对简单易懂。

1.1.1.4 下注

这部分的实现相对复杂一些。玩家下注的金额不仅要大于 LastBet，而且所下赌注需要保存下来，以便后续在整合返还多余资金时作为参考依据。

1.1.1.5 比牌

比牌操作首先要进行下注，随后选择参与比较的玩家。但要注意，所选玩家必须是尚未弃牌、仍在牌局中的玩家。
大致原理便是如此，不过要留意，比牌失败的玩家将退出牌局。

1.1.5 结束游戏的情况

1.1.5.1 只剩下一个人的情况

在此情形下，我们需要对存活人数进行实时监测。

1.1.5.2 剩下所有人 ALL_IN 的情况

我们知道，一旦有玩家选择 ALL_IN，其首个 ALL_IN 的位置值会被保存下来。当移动的下标再次回到首个 ALL_IN 值的位置时，意味着游戏结束。不过，这种结束情况与只剩一人的情况有所不同。由于每位玩家 ALL_IN 投入牌桌的金额各异，为确保公平，在进行大小比较之前，需将最小的 ALL_IN 值作为每位玩家实际投入的金额，同时将多余的资金返还给每位玩家（已退出牌局的玩家除外）。
